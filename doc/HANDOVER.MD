## 本次完成

- 分析实拍原图与裁切预览：清晰度损失主要来自 900px 兼容缓冲降采样、透视阶段双线性混色以及第二次 JPEG 编码，不是边缘定位错误。
- 扫描兼容缓冲最长边由 900px 提高到 1280px；原分辨率离屏路径继续保持相机原始纸张像素。
- 文字文档透视改用最近邻锐利采样，直接复制原图像素，不再让黑色笔画与白色纸面混合成灰边。
- 像素复制使用 `Uint32Array`，同尺寸本机基准从双线性约 54ms 降至约 18ms，核心透视阶段约提升 3 倍。
- JPEG 输出质量由 0.98 提升到 1，减少再次编码导致的细字损失。
- 后台处理分片由每次 36 行提高到 96 行，显著减少定时器切换开销；离屏图片加载等待由 1.5 秒缩短到 0.8 秒。

## 真机复验重点

- 使用同一份小字号公文分别拍摄旧版和本版，放大到 100% 比较黑色笔画边缘，本版不应出现明显灰边和二次模糊。
- 比较从“正在裁切”到“裁切完成”的耗时；原分辨率透视阶段应明显缩短。
- 连续扫描 3 至 5 页，确认提高分片行数后界面仍可继续拍摄和滚动。
- 检查斜线、印章圆弧和小字号文字；最近邻采样应保持边缘清晰，不应出现不可接受的锯齿。

## 已知边界

- 原始照片自身失焦、抖动或微信转发压缩造成的模糊无法由裁切算法恢复。
- 最近邻采样优先保留文字边缘；在极端大角度、低分辨率照片上可能比双线性更容易看到轻微锯齿。
- 极低内存设备仍可能使用 1280px 兼容结果，以避免扫描失败；正常设备优先原分辨率离屏裁切。
- Sass 构建仍有上游 legacy API 与 `@import` 弃用警告，不影响当前生产包。
