# 签字大师防伪溯源实施

日期：2026-07-15

## 鉴定顺序

签字鉴别采用固定的两阶段流程，不再把系统来源和用户归属合并成一个结论：

1. 用户从微信文件或相册上传已经签署的 PDF、JPEG 或 PNG。
2. 先验证 signMaster V2 系统签章、载荷完整性和文件正文摘要。
3. 只有系统来源通过后，才验证载荷内每个签字签章是否属于当前本机用户。
4. 报告列出报告编号、两阶段结论、签发/鉴定时间、提取的签字数量、页码、位置和签名档案匹配结果。
5. 图片报告在原文件上叠加已提取的签字范围；PDF 显示从文件防伪载荷提取的签字证据。

## signMaster V2 文件证据

- 每次导出构造 `SMV2` 载荷，包含产品标识、导出时间、文件名、格式、文档 ID、用户摘要和每个签字的页码、最终坐标、签章摘要。
- 文件编码完成后计算未加签正文的 SHA-256，再使用应用级 HMAC-SHA256 对规范载荷签章。
- JPEG 使用 APP15 段，PNG 使用带 CRC32 的私有安全复制块，PDF 在 EOF 后写入兼容的签章记录。
- 鉴别时先移除证据容器并恢复原始正文，重新计算 SHA-256；复制合法载荷到另一文件或修改正文都会导致第一阶段失败。
- 当前用户归属使用本机匿名稳定 ID 摘要和用户密钥重新校验每个 `SM1` 签字签章，不只比较显示名称。

## 双层图片盲水印

- 图片导出继续使用 8x8 亮度块二维 DCT。
- 新增固定的 signMaster 系统来源水印，与当前用户水印使用不同中频系数，二者可独立提取。
- 用户水印仍包含用户摘要和签字签章摘要，块位置由用户密钥派生。
- V2 容器被图片平台重编码移除时，可降级使用系统水印和用户水印给出来源/归属结果。
- 降级水印不包含完整签字坐标，因此报告会明确提示无法定位 V2 签字证据，不虚构签字数量。

## 动态行为与现场复签

- 手写点记录坐标、时间、设备可用压力值，并生成 128 维行为向量。
- 保存签名使用本机密钥生成 `SM1` 防篡改签章。
- 文件鉴定与现场复签相互独立；用户可在文件报告后继续现场复签，也可单独检查本机签名档案。
- 现场复签输出形态、书写节奏和综合一致性分数。

## 用户身份与数据边界

- 用户 ID 位于 `wx.env.USER_DATA_PATH/sign-master/user_id.txt`，不使用 OpenID、云函数或业务接口。
- 上传的待鉴定文件、文件摘要、签字证据和报告只在本机处理，不发送到服务器。
- 清除微信小程序完整数据或卸载后，本机 ID 和密钥会被删除；外部文件仍可验证系统来源，但不能再认定属于新的本机用户。

## 安全边界

- 这是纯前端离线方案。应用级系统签章密钥随客户端分发，可检测常规篡改、载荷移植和非 signMaster 文件，但不能替代服务端证书、可信时间戳或司法电子签名体系。
- 截图、强裁剪或大幅重新压缩可能只保留部分盲水印；界面不会把“未检测到”表述为确定伪造。
- 2026-07-15 之前导出的图片可使用旧版当前用户 DCT 水印降级鉴别；旧 PDF 没有 V2 正文绑定证据，不能独立通过系统来源鉴定。

## 自动化验证

- PDF、JPEG、PNG 三种 V2 容器嵌入、提取和原正文恢复。
- 文件正文被修改后，系统签章仍存在但正文摘要失败。
- 合法系统文件切换到另一用户时，系统来源通过、用户归属失败。
- 系统和用户两层 DCT 水印同时嵌入后均可独立恢复。
- H5 真实上传已签署 JPEG，报告严格先显示系统来源，再显示当前用户归属，并提取一个签字位置。
- 微信生产包 18 个页面、4 个 Tab 和签字鉴别分包加载通过。
